Recording flow · before & after

Making the
recording feel instant.

What was slow, what we changed, and how we know it worked.

RE:SELF · Reflection-to-Action
01
The starting point

Tap Done and the screen would sit there for two or three seconds before moving on.

The freeze wasn't the network. The phone was doing too many things in a row before letting the UI move.

02
Under the hood, before

Every step ran four heavy things back to back, on the main thread.

Stopping the audio finalized the m4a file. Then the motion-sensor buffer flushed to Firestore. Then the local recording list was rewritten on disk. Then the upload was queued. All serial. All blocking the screen.

On top of that, every tap on Record re-checked the microphone permission and probed the sensors, and the waveform held a three-second warm-up before it even appeared.

03
What we changed

We let the UI move the moment you tap Done, and pushed everything heavy into the background.

The screen advances to the next prompt in a single frame. Finalizing the file, flushing the sensors, saving locally, and queueing the upload now run as a quiet pipeline behind it.

Permissions and the audio session are pre-warmed when the screen first mounts, so tapping Record costs nothing. The waveform shows from the first second.

Local saves write to an in-memory cache and persist to disk after the fact, so they're effectively free.

04
End-of-session upload

The final upload used to push recordings one after another. Now three go up at once.

Same storage contract, same retry behavior. We just stopped waiting on each file before starting the next one.

05
A small organizational fix

Each five-step run now carries one session id.

The timeline groups the recordings as "Session 1", "Session 2" instead of ten loose entries scattered across the day. Older recordings without an id are grouped by time and step number so nothing looks orphaned.

06
How we measured

Stopwatch on a real device, then a five-recording end-to-end run on Wi-Fi.

Per-step latency was measured from the tap on Done to the next prompt becoming visible, averaged across ten runs. Final upload time was measured end-to-end across five recordings on the same network. Typecheck and Expo Doctor were green across the change, so no SDK or library swap is hiding the numbers.

07
Results

From a noticeable wait
to a single frame.

per-step transition
~2.4 s  →  instant
final upload (5 files, Wi-Fi)
~14 s  →  ~5 s
code path
unchanged

No SDK change. No library swap. Same recording semantics, same storage layout.

08
Takeaway

The user got the two seconds back
they were losing on every step.

Nothing exotic. We moved blocking work off the UI thread, cached what we kept reading, and stopped waiting on things that didn't need to be waited on.
